## 🛠️ 2. 터미널 및 환경 설정 트러블슈팅 기록

- **Windows 개발자 모드 오류 해결**: 플러그인 빌드 시 발생하는 symlink(심볼릭 링크) 오류 방지를 위해 OS 개발자 환경 켜기 숙지.
- **npm PowerShell 보안 오류 해결**: `.ps1`이 막혔을 때 뒤에 `.cmd`를 붙여서 `npm.cmd install`로 관리자 권한 없이 쉽게 설치하는 팁 체득.
- **npm SSL 인증 오류 우회**: 사내망이나 학교망/프록시망에서 `npm.cmd config set strict-ssl false`로 임시 우회하여 패키지를 성공적으로 다운로드하는 방식 해결.

## 🌿 3. 깃(Git) 브랜치 심화 학습

### ① 브랜치(Branch)란? 왜 쓰는가?
- 본체인 `main` 브랜치를 오류로부터 안전하게 지키기 위한 일종의 **병렬 세계(평행 우주)**.
- 이 우주에서 에러가 나거나 코드를 다 엎어도, 본체의 코드는 100% 안전하다.
- **작업 원리**: 빈 폴더에 파일만 넣는 게 아니라, `main`의 전체 소스 코드 100%가 똑같이 복사되어 들어간다. 평소처럼 앱 전체 폴더를 보면서 수정하면 되고, 나중에 합칠 때 방금 내가 "변화시킨" 부분만 똑 떼어내어 본체에 스며들듯 합쳐진다(Merge).

### ② 상황별 꿀팁 명언록
> "개발 혼자 하는데 귀찮게 브랜치 파지 말고 그냥 main만 쓰면 안되나요?"
- **물론 됩니다!** 오타 수정 같은 자잘한 작업은 그냥 `main`에 올리는 게 빠릅니다.
- 하지만 "카카오맵 연동" 같은 앱을 멈추게 할 수 있는 **큰 공사를 할 땐 꼭 브랜치를 만들어야 나중에 피눈물을 흘리지 않습니다.**

### ③ 기능 개발 브랜치 사이클 (GitHub Flow 기준)
아래 명령어 4단계를 항상 머릿속에 담아두기!

1. **평행 우주 만들기 (본체 코드 100% 복사됨)**
   ```bash
   git checkout -b feature/네이밍
   ```
2. **신나게 앱 고치고 세이브(Commit) 치기**
   ```bash
   git add .
   git commit -m "feat: 이런 거 저런 거 고쳤음"
   ```
3. **내 우주를 깃허브에 백업(Push)하기**
   ```bash
   git push origin feature/네이밍
   ```
   (이후 깃허브에 들어가서 초록색 **Pull Request(PR)** 버튼을 누르면 본체에 안전하게 합쳐짐!)

4. **다시 메인 세계선으로 귀환 🚀**
   ```bash
   git checkout main
   git pull origin main
   ```
   (위의 PR 과정 후, 깃허브에서 다 쓴 브랜치는 헷갈리지 않게 삭제버튼을 눌러주는 것이 좋다)
